üí≠ Answers to Reflection Questions

Q1: How is storing data in a JSON file different from using browser storage?

JSON file storage persists on the server/backend side and can be modified by server-side scripts like Python. Browser storage (localStorage, sessionStorage) is client-side only and limited to 5-10MB per domain. JSON files allow structured, portable data that can be shared, backed up, and manipulated programmatically. Browser storage is ideal for user preferences, while JSON files enable collaborative data management.

Q2: How did you use Python to create or update your JSON file?

I created save_entry.py that imports the json module and os module. The script loads existing reflections.json, appends new entries with timestamps and metadata, and writes the updated data back. Each reflection object contains date (ISO format), text, category, and unique ID. The script provides an interactive menu for users to add new reflections or view existing ones, demonstrating CRUD operations on JSON files.

Q3: What does your PWA show locally vs on GitHub? Are they the same? Why or why not?

Locally, the PWA displays all features including dynamically loaded JSON reflections with real data. On GitHub Pages, users will see the HTML structure and styling, but the JSON fetch may fail due to CORS policies when loading backend/reflections.json from GitHub. This limitation demonstrates why full-stack applications often require hosted backends. However, the GitHub version still displays the page structure and is fully functional for learning demonstration purposes. Static JSON files work, but dynamic Python updates won't execute in GitHub's static hosting environment.

Q4: What extra feature did you add to your PWA using the JSON file, and why?

I implemented a CSV export feature that allows users to download their reflections as a spreadsheet. This feature adds significant value because: (1) it enables data portability and archival, (2) users can import data into analysis tools or documentation platforms, (3) it demonstrates advanced JavaScript concepts like blob creation and download triggering. I also built a comprehensive statistics dashboard that calculates metrics from the JSON data in real-time, providing users with insights about their reflection patterns and writing habits. Both features enhance the PWA's utility beyond simple display functionality.

üîë Understanding Frontend-Backend Connection

The connection between frontend and backend is vital for creating an application that is usable over time and across multiple devices. The frontend is the quick, responsive part that users interact with, but it doesn't have the memory needed for a journal. The backend, running Flask on the server, is where the application's state and data persistence live. It guarantees that the entries I write today will still be there a month from now, even if I use a different laptop. This fundamental separation of concerns‚ÄîUI on the client and data management on the server‚Äîis what defines a modern web app.

üåê HTTP Methods in Flask Implementation

I used GET, POST, and PUT methods for my application's logic:

  • GET: Used simply to fetch the current list of reflections. It's safe and doesn't modify the server's data. This method retrieves all reflections from the JSON file and returns them as JSON to the frontend.
  • POST: Used every time I submitted a new journal entry. This method correctly signals to the server that the request is intended to create a new record in the JSON file. The Flask route accepts JSON data, validates it, appends it to the reflections list, and saves the updated file.
  • PUT: Used for my editing feature. I used it specifically to replace an existing record with a completely new version of the data, which is the standard purpose of the PUT method. The route identifies the reflection by ID, updates all fields, and saves the modified list back to the JSON file.
  • DELETE: Implemented for removing reflections. This method finds the entry by ID, removes it from the list, and persists the changes.

üîÑ Flask vs Browser JSON Storage

The main difference is that using Flask gives us a secure way to write data. When I read a JSON file directly in the browser, I'm just getting a local copy of the information; there's no way to edit the file on the server. If I used a client-side method like localStorage, the data would be tied only to my specific browser's memory.

Flask mediates the entire process. It hosts the data file, and my JavaScript can send small API requests to Flask's routes. Flask, being server-side, has the permissions to access the server's file system, making the necessary changes to the reflections.json file and saving them securely. This approach enables:

  • Persistent data storage across devices and browsers
  • Secure write operations that client-side JavaScript cannot perform
  • Centralized data management on the server
  • Ability to share and access data from any device
  • Proper separation of concerns between presentation and data layers

üêõ PythonAnywhere Deployment Challenges

My biggest headache came from an initial data crash during the first run. Since the reflections.json file didn't exist when I started my web app for the first time, or if it was empty, my Python code would throw a JSONDecodeError when it tried to load the file contents. The whole server would crash because it was expecting data and found nothing.

I handled this by updating my Python functions (load_reflections). I wrapped the part that reads the JSON file in a try...except block. If the JSONDecodeError occurred, instead of crashing, the function now gracefully returns an empty Python list ([]). This allows the application to start cleanly and simply create the very first reflection entry without crashing.

Solution Code Pattern:

def load_reflections():
    if os.path.exists(REFLECTIONS_FILE):
        try:
            with open(REFLECTIONS_FILE, 'r', encoding='utf-8') as f:
                return json.load(f)
        except (json.JSONDecodeError, IOError):
            return []
    return []

‚ú® Extra Feature: Edit Entry Functionality

I chose to implement an Edit Entry feature, which allows a user to modify any of the reflections they've previously submitted.

I implemented this using a PUT request. The frontend sends the entire updated reflection object to a specific Flask route along with the entry's ID. In my flask_app.py, the code finds the old entry using the ID, replaces it with the new data, and then saves the complete updated list back to the reflections.json file. I added this because journal entries often require small corrections or expansions after they're first written. Having the ability to update the data permanently on the server was a necessary step for making the PWA truly functional.

Technical Implementation:

  • Frontend JavaScript populates the form with existing reflection data when edit button is clicked
  • Form submission detects edit mode via a data attribute (dataset.editId)
  • PUT request sent to `/api/update_reflection/<id>` with updated JSON payload
  • Flask route locates the reflection by ID, updates all fields while preserving timestamp and ID
  • Updated list is saved back to JSON file, ensuring data persistence
  • Frontend refreshes to display the updated reflection

üõ†Ô∏è Technical Architecture

The Flask application follows RESTful principles with clear separation of concerns:

  • API Routes: `/api/reflections` (GET), `/api/add_reflection` (POST), `/api/update_reflection/<id>` (PUT), `/api/delete_reflection/<id>` (DELETE), `/api/search_reflections` (GET with query parameter)
  • Data Layer: JSON file storage with helper functions for loading and saving reflections
  • Frontend Layer: Vanilla JavaScript using Fetch API for all HTTP requests
  • Error Handling: Comprehensive try-catch blocks with meaningful error messages returned as JSON
  • Data Validation: Server-side validation ensures required fields (title, content) are present before saving
